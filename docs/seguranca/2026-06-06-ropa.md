# ROPA — Registro das Operações de Tratamento de Dados Pessoais

**Sistema:** IFP Connect · **Controlador:** Instituto Família Pôncio (IFP) · **Versão:** 1.0 · **Data:** 2026-06-06

> Documento exigido pela **LGPD Art. 37** (registro das operações de tratamento). É um **catálogo técnico vivo**, derivado do modelo de dados real do sistema (`prisma/schema.prisma`) e dos controles verificados na [revisão LGPD leve](./2026-06-05-lgpd-revisao-leve.md). **Não é parecer jurídico** — as bases legais e prazos de retenção marcados com ⚖️ devem ser **confirmados com o Encarregado (DPO)/assessoria jurídica** antes do uso com dado real.

## 2. Operações de tratamento (por finalidade)

Cada bloco é uma operação. Categorias **sensíveis** (LGPD Art. 11) marcadas com 🔴.

### 2.1 Cadastro do cidadão e composição familiar

| Campo                   | Valor                                                                                                                                     |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **Finalidade**          | Identificar o beneficiário e o núcleo familiar para o atendimento institucional                                                           |
| **Categorias de dados** | Nome completo, nome social, CPF, data de nascimento, contato (telefone/endereço), composição familiar (`Cidadao`, `Familiar`, `Endereco`) |
| **Base legal** ⚖️       | Consentimento do titular/responsável (Art. 7º, I) — termo físico assinado, registrado em `Consentimento(tipo=tratamento_dados)`           |
| **Titulares**           | Cidadãos e familiares (inclui menores → consentimento do responsável legal)                                                               |
| **Retenção** ⚖️         | Enquanto houver vínculo ativo + período a definir; ao encerrar, **anonimização** (`anonimizadoEm`, server action de anonimização)         |
| **Destinatários**       | Staff interno conforme RBAC (recepção, social, gestão, super_admin)                                                                       |

### 2.2 Triagem socioeconômica / assistência social

| Campo                   | Valor                                                                                                                    |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **Finalidade**          | Avaliar vulnerabilidade e direcionar o cidadão aos programas do instituto                                                |
| **Categorias de dados** | 🔴 Dados socioeconômicos (renda, benefícios, situação de vulnerabilidade), encaminhamentos (`Triagem`, `Encaminhamento`) |
| **Base legal** ⚖️       | Consentimento (Art. 7º, I) e/ou proteção do titular (Art. 11, II, "e", quando aplicável a programa assistencial)         |
| **Titulares**           | Cidadãos em triagem                                                                                                      |
| **Retenção** ⚖️         | Vinculada ao histórico assistencial; anonimização ao encerrar vínculo                                                    |
| **Destinatários**       | `social` (transversal por papel), gestão, super_admin                                                                    |

### 2.3 Atendimento médico (consultas, prontuário, documentos)

| Campo                   | Valor                                                                                                                                                              |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Finalidade**          | Prestação de cuidado em saúde (agenda, consulta, evolução clínica, prescrição)                                                                                     |
| **Categorias de dados** | 🔴 Dados de saúde: prontuário/notas clínicas assinadas, CID, sinais vitais, receitas, atestados, encaminhamentos (`Consulta`, `Prontuario`, `Receita`, `Atestado`) |
| **Base legal** ⚖️       | **Tutela da saúde, por profissionais de saúde** (Art. 11, II, "f") — base primária; consentimento (Art. 11, I) registrado como reforço                             |
| **Titulares**           | Pacientes atendidos na unidade médica                                                                                                                              |
| **Retenção** ⚖️         | **Prontuário: retenção legal mínima** (referência CFM Res. 1.821/2007 — 20 anos). **Não anonimizar antes do prazo legal.**                                         |
| **Destinatários**       | Profissionais de saúde e gestão da unidade (`podeVerSaudeCidadao`); **não** recepção, **não** social                                                               |

### 2.4 Capacitação profissional (matrícula, frequência, certificado)

| Campo                   | Valor                                                                                                                                                            |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Finalidade**          | Gerir turmas, presença e emissão de certificado                                                                                                                  |
| **Categorias de dados** | Matrículas, registro de frequência, certificados emitidos (`Matricula`, `Presenca`, `Certificado`)                                                               |
| **Base legal** ⚖️       | Consentimento (Art. 7º, I) e execução de atividade do instituto                                                                                                  |
| **Titulares**           | Alunos das capacitações                                                                                                                                          |
| **Retenção** ⚖️         | Certificado e histórico de conclusão: retenção prolongada (validade/verificação pública via código). Frequência: vinculada à turma                               |
| **Destinatários**       | Instrutor (travado na própria turma), secretaria/gestão; **verificação de certificado é pública** (rota `/verificar/[codigo]`, expõe só nome + curso + validade) |

### 2.5 Uso de imagem

| Campo                   | Valor                                                                                                                 |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **Finalidade**          | Divulgação institucional, prestação de contas e comunicação                                                           |
| **Categorias de dados** | Autorização de uso de imagem com **escopo granular**: interno, redes sociais, imprensa (`Consentimento(tipo=imagem)`) |
| **Base legal**          | **Consentimento específico e destacado** (Art. 7º, I) — granular e **revogável por escopo** a qualquer tempo          |
| **Titulares**           | Cidadãos fotografados/filmados                                                                                        |
| **Retenção**            | Enquanto vigente o consentimento; revogação registrada em `revogadoEm`                                                |
| **Destinatários**       | Comunicação/gestão; uso externo só nos escopos autorizados                                                            |

### 2.6 Anexos / documentos digitalizados

| Campo                   | Valor                                                                                             |
| ----------------------- | ------------------------------------------------------------------------------------------------- |
| **Finalidade**          | Guardar documentos do cidadão (comprovantes, laudos)                                              |
| **Categorias de dados** | 🔴 Pode conter dados sensíveis (documentos de saúde) (`Anexo` em MinIO)                           |
| **Base legal** ⚖️       | Acompanha a finalidade do anexo (saúde → Art. 11; cadastral → Art. 7º)                            |
| **Retenção** ⚖️         | Soft-delete (`deletedAt`); remoção física no fluxo de anonimização                                |
| **Destinatários**       | RBAC com unit-scope; entrega via **URL presigned** (tempo limitado, sem acesso direto ao storage) |

### 2.7 Contas de acesso (staff)

| Campo                   | Valor                                                                                                |
| ----------------------- | ---------------------------------------------------------------------------------------------------- |
| **Finalidade**          | Autenticar e autorizar colaboradores no sistema                                                      |
| **Categorias de dados** | Nome, e-mail, **senha (hash bcrypt)**, papéis e escopo de unidade (`User`, `Role`)                   |
| **Base legal**          | Execução de contrato/relação com o colaborador + legítimo interesse (controle de acesso e segurança) |
| **Titulares**           | Profissionais e colaboradores do IFP                                                                 |
| **Retenção**            | Enquanto vínculo ativo; desativação em vez de exclusão (preserva trilha de auditoria)                |
| **Destinatários**       | super_admin (gestão de usuários)                                                                     |

### 2.8 Painel de chamada na TV (exibição + locução de nome)

| Campo                   | Valor                                                                                                                                                                                                      |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Finalidade**          | Chamar o cidadão para atendimento numa TV em área semipública, com acessibilidade (nome exibido em destaque **e falado por voz** — público com baixa visão / que não lê)                                   |
| **Categorias de dados** | Nome do cidadão (snapshot de `nomeSocial` ?? `nomeCompleto`) + destino (origem da chamada) (`Chamada`)                                                                                                     |
| **Base legal** ⚖️       | Execução do atendimento + acessibilidade (legítimo interesse do titular em ser atendido). Exibição de nome em área semipública é decisão explícita de acessibilidade do instituto                          |
| **Titulares**           | Cidadãos chamados (médico; triagem)                                                                                                                                                                        |
| **Retenção**            | Evento de chamada (`Chamada`) — log operacional; sem prazo definido ainda (curto, descartável)                                                                                                             |
| **Destinatários**       | Quem estiver na sala de espera (semipúblico, por natureza da chamada). A **tela da TV e o endpoint de polling exigem sessão de quiosque** unit-scoped (papel `painel`) — sem exposição pública pelo Funnel |
| **Mitigações**          | Usa **nome social** quando houver; a chamada é **auditada** (`paciente_chamado`); a TV loga como usuário de quiosque mínimo (só vê o painel, não acessa fichas); o vídeo/anúncios não contêm dado pessoal  |

## 3. Medidas de segurança (verificadas no código)

- **Superfície pública mínima** (`src/proxy.ts`): tudo que tem dado pessoal exige sessão.
- **RBAC por papel + unidade** (`canAccessUnidade`, `podeVerSaudeCidadao`, `podeVerSocioCidadao`) — enforcement na camada de dados, não só no JSX.
- **Troca de senha forçada no 1º login** + senhas demo neutralizadas (`mustChangePassword`).
- **Trilha de auditoria** append-only (`lib/audit`) — inclui registro/revogação de consentimento; atos clínicos sem bypass de admin.
- **Soft-delete** (`deletedAt`) + **anonimização** real (mascara cidadão/familiar/endereço + remove anexos).
- **Em trânsito:** HTTPS (TLS). **Anexos:** URL presigned. **Senhas:** bcrypt.
- **Backup** diário com rotação (`ops/vm/backup.sh`); cifragem do dump em andamento (item #1 da revisão leve).

## 5. Mecanismo de consentimento no sistema

O aceite do **termo físico assinado** é registrado na ficha do cidadão (seção "Consentimentos (LGPD)", gated por `podeGerirConsentimento`):

- **Tratamento de dados** — versionado (`VERSAO_TERMO_TRATAMENTO`); registrável/revogável; status "desatualizado" quando o termo muda de versão (pede re-registro).
- **Imagem** — granular (interno/redes/imprensa), versionado e revogável por escopo.
- Cada registro grava **versão + data + quem registrou** e gera evento de auditoria (`consentimento_registrado` / `consentimento_revogado`).

> Atualizar este ROPA sempre que: (a) entrar nova categoria de dado, (b) mudar uma finalidade/base legal, (c) entrar um novo operador/suboperador (ex.: serviço de nuvem, WhatsApp API).
