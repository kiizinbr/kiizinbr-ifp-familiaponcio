# IFP Connect MVP — Plano 2: RBAC + Audit Log + RLS — Implementation Plan (DRAFT)

> **Status:** DRAFT — escrito sem input do Erick durante autopilot overnight 2026-05-24. Tarefas e estrutura prontas; **decisões abertas (seção 0) precisam ser resolvidas antes da execução**.

**Goal:** Adicionar controle de acesso baseado em papéis (RBAC), audit log de eventos sensíveis, e Row-Level Security (RLS) no Postgres, transformando o app de "qualquer usuário logado vê tudo" em "cada papel vê só o que pode".

**Architecture:** Modelo many-to-many `User <-> Role` (com `unitScope` opcional para roles por unidade). Permissões derivadas do papel (não armazenadas). Audit log como tabela append-only com retenção configurável. RLS no Postgres via policies SQL que leem o `app.current_user_id` setado pelo Prisma a cada query.

**Tech Stack adicional:** `pgTAP` para testes de RLS, helper `withAuthContext()` para setar `app.current_user_id` no início de cada Prisma transaction.

**Plano base:** `docs/superpowers/plans/2026-05-23-ifp-connect-mvp-01-foundation.md` (✅ done)

### 0.3 Audit log: o que registrar

Eventos candidatos:

- [ ] Login bem-sucedido
- [ ] Login falho
- [ ] Logout
- [ ] Criação/edição/exclusão de Ficha Cidadã
- [ ] Acesso de leitura à Ficha Cidadã (READ event — caro mas LGPD pede pra dados sensíveis)
- [ ] Mudança de role de outro usuário
- [ ] Export de dados (CSV, PDF)
- [ ] Acesso a dados médicos (subset de Ficha Cidadã)
- [ ] Mudança de status (aprovação, reprovação)

**❓ Pergunta:** Quais marcar? Padrão sugerido: tudo exceto "Acesso de leitura" (custo de I/O alto) **mas** sim incluir "Acesso a dados médicos" (CFM 1.821 pede).

### 0.4 Audit log: retenção

- Opção A — **Para sempre** (compliance LGPD: 5+ anos)
- Opção B — **Particionado mensal**, manter 24 meses, depois arquivar em S3/MinIO
- Opção C — **Hard delete** após 2 anos

**❓ Pergunta:** A, B ou C? Recomendado: B (LGPD pede acesso, mas particionar evita explosão de tamanho).

### 0.5 RLS granularity

- **Por unidade**: gestor_unidade só vê fichas cujo `unit_id` está no array `user_units` do JWT
- **Por cidadão**: profissional só vê fichas cujo `created_by` é ele OU se está em `cidadao_caregivers` (lista de profissionais autorizados)
- **Por dado sensível**: campos médicos (CID, prontuário) com policy separada — só `profissional` da unidade médica + `super_admin` veem

**❓ Pergunta:** Vamos começar com unidade + cidadão? Por dado sensível entra no Plano 4 ou já entra agora?
