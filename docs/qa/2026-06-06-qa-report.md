> **A história do QA, em uma frase:** o RBAC do IFP valida o **papel** (`canAccessUnidade`/`hasAnyRole`) mas, na maioria das Server Actions, **não valida o objeto** — o `id` (consulta, matrícula, cidadão, anexo, triagem...) vem do FormData do cliente e é usado direto na query. Resultado: um usuário legítimo de uma unidade alcança/modifica dados de **outras unidades e outros pacientes**. É a pegadinha "Server Action não herda o gate" — agora provada como **sistêmica** (médico, capacitação, cidadãos, social).

## 🔴 CRITICAL (10) — corrigir ANTES de qualquer uso com dado real

> Padrão único na maioria: **resolver o `id` do FormData → checar só papel → mutar/ler sem confirmar dono/unidade.** Fix base: após resolver o id, validar `canAccessUnidade`/`unitIdOrigem`/`turmaId` do **objeto** antes de agir.

1. **IDOR — check-in de consulta cross-tenant** · `src/app/medico/consultas/[id]/checkin-action.ts:25`
   `marcar/desfazerCheckinAction` faz `db.consulta.update({where:{id}})` com `id` do FormData; só checa role `recepcao`. Recepção do médico marca check-in de consulta de **qualquer unidade** (PHI). → validar `consulta.profissional.unitScope==='medico'` antes do update.

2. **IDOR — cancelar encaminhamento system-wide** · `src/app/medico/encaminhamentos/actions.ts:21`
   `cancelarEncaminhamentoAction` cancela qualquer `encaminhamentoId` (sem checar dono/unidade). Bloqueia encaminhamento de qualquer paciente. → checar a consulta de origem pertence ao médico.

3. **IDOR — presença sem validar turma** · `src/app/capacitacao/actions.ts:320`
   `registrarPresencasAction`: o `roster` (CSV de matriculaIds) não é validado contra `turmaId` → grava presença em matrículas de **outras turmas**, corrompe frequência/certificado. → `findMany({where:{id:{in:roster}, turmaId}})`.

4. **IDOR — transição de matrícula sem validar turma** · `src/app/capacitacao/actions.ts:154`
   `transicionarMatriculaAction` busca `matricula` só por `id` (sem `turmaId`) → confirma/cancela matrícula de qualquer turma. Blast radius = todas as 1064+ alunas. → `findFirst({where:{id, turmaId}})`.

5. **IDOR — emissão fraudulenta de certificado** · `src/app/capacitacao/actions.ts:363`
   `emitirCertificadoAction` emite certificado (`IFP-<hex>` público) pra matrícula arbitrária concluída ≥80%, sem validar `turmaId`. → mesmo fix do #4.

6. **IDOR — consentimento LGPD cross-tenant (write)** · `src/app/app/cidadaos/[id]/consentimento-actions.ts:19`
   As 3 actions de consentimento leem `cidadaoId` do FormData; `gate()` só checa role `podeGerirConsentimento`. Recepção de uma unidade registra/**revoga consentimento LGPD** de cidadão de outra. → carregar cidadão + `canAccessUnit(unitIdOrigem)`.

7. **Vazamento de PII — triagens de todas as unidades** · `src/lib/triagem.ts:55`
   `listTriagensPendentes`/`countTriagensAbertas` filtram só `status:'aberta'` (sem filtro de unidade); role `social` é global → todo assistente social vê nome/nome social/unidade de **todas** as unidades. → aplicar `unitFilter` por `getUserUnits(session)` (igual `getCidadaoStats`).

8. **IDOR — leitura de qualquer triagem** · `src/lib/triagem.ts:28`
   `getTriagem(id)` checa só role social → qualquer triagem por UUID (identidade do cidadão + do assistente + elegibilidades). → validar `cidadao.unitIdOrigem` ∈ unidades do caller.

9. **IDOR — consentimento LGPD cross-tenant (gate)** · `src/app/app/cidadaos/[id]/consentimento-actions.ts:10`
   Mesma raiz do #6 visto pelo `gate()` — role-only, sem unit scope. (consolidar fix com #6.)

10. **Destrutivo cross-tenant — remover anexo** · `src/app/app/cidadaos/[id]/anexo-actions.ts:168`
    `removeAnexo` carrega `cidadao.unitIdOrigem` mas só checa `hasAnyRole('super_admin','gestor_unidade')` — nunca usa o unitScope. `gestor` de uma unidade **apaga anexo (MinIO + DB)** de cidadão de outra (perda irreversível). → `canAccessUnit(anexo.cidadao.unitIdOrigem)`.
